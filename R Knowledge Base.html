Welcome to  Ultidoc, the Ultimate Document

and the Ultimate Document Processor



Doug Skuce


July, 2017


This is an experimental site whose purpose is explore a new way of constructing documentation of any kind, not just technical.  Examples could range from school texts to repair manuals to tax laws.  We call such a document an Ultidoc. (It took many hours to find a catchy name that wasn't taken!) You are now reading the first Ultidoc!   And then you can try the processor.


NBNBNB!!!   This site and the associated software are at a very early stage of development, so don't expect everything to be perfect.  Also, the emphasis has been on getting the system minimally working, so that there are many things wrong or missing with both it and the example knowledge base.


We will use the language R as an example topic, partially because we have experienced many slow hours of searching the web for details of the language as we learned it in order to implement a processor for Ultidocs.  If a document such as this had existed, ie an Ultidoc for R,   we could have found many of these answers instantly. Not only that, we could easily spot conflicting, vague, and missing information. Hopefully, some day all documentation will benefit from this technology. Studies have shown that programmers spend a major portion of their time seeking information, which is usually hard to find and often confusing or incomplete. So Ultidocs have two main advantages: as a better way to find information, and as a better way to author it. Authoring can be done collaboratively on the web.


Ultidoc has the following design principles:


  • In the future, all documentation should be HTML based. No more PDFs, CHM, or Word, etc.
  • Ultidocuments will  conform to certain universal structural standards, both to facilitate human understanding and machine processing, somewhat like Wikipedia, particularly its infoboxes, which can be semantically processed. https://en.wikipedia.org/wiki/Help:Infobox. One way to describe Ultidoc would be "the future Wikipedia", help systems, and web-based information of any kind.
  • It should facilitate document creation by allowing the authors to organize information better, particularly in achieving consistency and correctness.
  • This is done with a semantic processor, the  Ultidoc processor which can scan a document and extract certain structured information, both for later retrieval and for correctness analysis. In a similar manner, WP's infoboxes can be automatically converted to RDF, but RDF is both limited in expressivity and lacks friendly  means for processing it. But Utilidoc uses something much more powerful and flexible, something we call wordsheets, a kind of table, a spreadsheet for words.
  • Wordsheets are embedded in documents; they are simply HTML tables in which strict linguistic rules must be followed to permit machine processing.



If you follow enough formal rules, it is possible to convert an Ultidoc into an OWL ontology or any other formal knowledge structure.

But you dont have to follow the rules if you do not want that degree of precision, and you can create new ones.
