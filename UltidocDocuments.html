Ultidoc Documents

A Ultidoc document is a static web site (desktop or published on the web) built according to certain principles. A UD page is a normal web page that may contain certain markup in tables. Any tool that can create a site may be used.


The most important markup is done with HTML tables which we call wordsheets (a spreadsheet for words). Each column contains data all of the same semantic type like database or spreadsheet tables.  The most common table is the standard table, but a user is free to invent any other table structure, but then they must write code it they wish to process it.

NB:

  • the top row here is not part of the table. It names the columns, 10 in our standard table (only 6 are used at the moment).  All tables (of a given type)  have the same structure, else they could not be treated equally. This is our standard structure, useful for most applications. A table's top left cell must begin with !, It usually is the topic of the page it is on, which is the page's name. This is how the processor ignores any other tables on the page. There can be several tables on one page.
  • # is a shorthand for the the top left cell, the topic, here 'cat'.
  • each row can be about any topic, though most of the time it should be about the page topic.
  • "isako" is short for "is a kind of".  It will cause property inheritance (see below).  Thus MyCat, if declared 'isa' ("is an instance of") will inherit  love fish and hate dog/most
  • each entry can be followed by appropriate modifiers, discussed later
  • source identifies the page url; it is a link to see what else is on the page.
  • there are many rules one can invent for the various modifiers (eg following the / or in the mods column). We are working on these. Quantifiers, adjectives and adverbs are put in strict places.
  • Being HTML, is is very easy to assemble a page from snippets copied from existing pages. This has been used heavily in the R pages.



Certain defaults apply:

  • the subject is universally quantified A, ie if there is no quantifier it is assumed to be "every". This is inserted with a warning.
  • the object is existentially quantified E, ie it is assumed to be "1 or more"


Usually a property will apply to a sub, but with more restrictions. For example a universal quantifier may be narrowed to many, or a numeric  quantifier like  1:3  (ie one, two or three) may be narrowed to 1:2. It is possible to change something in a sub that contradicts, e.g. if we say no dog likes cats and MyDog likes cats, or change the quantifier to 1:4. Any such identifiable contradiction can be detected by the processor. The author can either fix it or flag it as a known contradiction.


A predicate normally inherits to subkinds or instances. But if there is a flag X (after the/)  it will inherit without the object and mods, ie only subject and predicate will inherit.  An example would be maybe 'location. If something has a location, its subs also have a location, but it may be unrelated, so probably it should inherit blank. Then a sub can have its own value for its location.  If there is a flag C it will inherit to subkinds but not to instances. An example might be 'date of invention'.  While any type of car has a date when it was invented, my car was not invented. Thus inventing is an unusual property that does not apply to instances but only to kinds. Another would be "number of instances"


At the moment (July 2017) the processor does inheritance but no other processing. See Ultidoc processor

The next step will be consistency checking like we just described.


The remaining columns have other future purposes.


Pages should be arranged in hierarchies, but if certain rules are followed, parts of the hierarchy that conform have desirable logical consequences. When making a subpage, one must decide between four possible subpage relations to the parent page:


a subtopic of                (isasto)                restaurants could be a subtopic of New York City

a kind of                        (isako)                a car is a kind of vehicle

an instance of                (isa)                        my car is an instance of a car;  3 is an instance of a number

a part of                        (isapo/hasp)        my left hand is a part of me


Subtopic is the default, and carries no logical meaning. The other three must be specified and participate in what is called property inheritance, where subpages inherit certain information from their ancestors. Thus a subtopic usually does not get any information from its parent. In other words, it is simply an organizational structure, like file folders. But the other three can inherit facts from their ancestors, ie if you wish, you can ask "show me all statements that are true of subject S because they are true of one of S's ancestors". So if


"integers can be added",

       then knowing

"1 isa integer"

then we know

"1 can be added",


ie if something is an integer it can be added.


To decide between kind and instance, ask the question "is X a kind of Y?"  If you say no or this seems odd, it probably is an instance.  If  the question "what kinds does X have" seems odd, X is probably an instance. A kind describes properties or attributes of its instances, if any exist. Thus:


"1 isa integer"

is true but

"1 isako integer"

is not, because 1 is not a kind. A "kind" is a category of things that belong to it. 1 is not a category; you can't say  "something is a 1". (You can say "this is a 1" but you mean "this symbol means 1"


A part may inherit some information, such as ownership or location. (if you own a car you own any part of it; if your car is in the garage, so are its wheels, assuming they are on it!).


Thus topics and their pages should be arranged in hierarchies, with subpages describing the same topic but confined to some narrower or closely related aspect, Thus under vehicle you could have cars, but not car manufacturers, which should go under manufacturers. Pages describing an instance, like New York City, can only have part or subtopic  subpages, since there are no "kinds of" New York City. NYC is an "instance of" a city. There are not multiple NYCs. It is not a kind. Note that in Wikipedia, a page is either


about a kind eg "cities"

about an instance eg "NYC'

or something else, eg a list of cities


Normally a page should be confined to one clear topic, like a Wikipedia page. Note how any WP page can have subtopic, kind, or instance relation to others via the Category mechanism. These relations are not distinguished in WP. So you can put a restaurant in the category "NYC"; the Category mechanism has no formal rules and does not partake in any processing. You may chose if a subpage is to have the meaning "isako". Thus this would be wrong:


a page about NYC

a subpage about restaurants in NYC, with isako. What you need is to write



!NY restauants

isako
